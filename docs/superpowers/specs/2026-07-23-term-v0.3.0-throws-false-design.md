# @getpipher/term — SPEC (v0.3.0): `{ throws: false }` structured-result opt-out

**Status:** DRAFT → self-reviewed ✅ → pending RECTOR review (re-created after overlay loss)
**Date:** 2026-07-23
**Stack target:** tmux 3.7+. macOS + Linux. Node 20+.
**Parent spec:** `docs/superpowers/specs/2026-07-22-term-design.md` (v0.1.0).
**Scope:** Additive, non-breaking increment over v0.2.0.

---

## 1. Overview

v0.1.0/v0.2.0 ship a single contract for the wait family: `waitFor` and
`waitForQuiet` throw `TermTimeoutError` on timeout. The v0.1.0 spec listed a
non-throwing variant as a non-goal ("deferred to v0.2 if probe-then-branch
usage proves it"). Probe-then-branch IS a real agent pattern — a programmatic
importer captures, decides whether the pane is ready, and branches. Today that
pattern forces a `try/catch` around every `waitFor`, and a thrown error is the
wrong shape for "I polled and the pane isn't there yet" (an expected branch,
not an exceptional failure).

v0.3.0 adds an opt-in structured-result variant: `{ throws: false }` makes the
wait functions **return** a Result-style discriminated union instead of
throwing. The default (`throws: true`) is unchanged — zero breakage, opt-in
only. The `TermTimeoutError` instance is reused verbatim as the failure
payload, so no new error type and no information loss.

### Goals (v0.3.0)

- Add `throws?: boolean` to `WaitForOptions` and `WaitForQuietOptions`
  (default `true` → back-compat).
- When `false`, return `WaitForResult` instead of throwing.
- Reuse the existing `TermTimeoutError` as the failure value (same fields).
- One new exported type (`WaitForResult`); no new error class.
- **Lib API only** — direct importers (`@getpipher/vision` QA, harnesses) opt
  in; the pi tool surface is unchanged.

### Non-goals (v0.3.0)

- **No tool-surface change.** The tool's `execute()` already catches
  `TermTimeoutError` and returns it as a structured `isError` result — the
  LLM agent already gets a structured timeout and never sees a throw.
  `throws:false` is a **lib ergonomics** feature for programmatic importers.
  Wiring it through the tool would add schema surface + dispatch threading +
  an `Error`-to-plain-object normalization at the `JSON.stringify` boundary
  (Error subclasses serialize to `{}` — non-enumerable props). Not worth it;
  revisit if a concrete agent need arises.
- **No failure-reason expansion.** Combined-mode timeout stays
  indistinguishable from pattern-never-matched in the error. The error already
  carries `lastCapture`; the caller can inspect it. A `reason` field is YAGNI
  until usage shows it's needed.
- **No change to `throws: true` semantics** — identical to v0.2.0.
- **No DSL / composite actions** — deferred per the parent spec.
- **No panel / `/term` slash** — still agent-primary.

## 2. Decisions (locked during brainstorm)

| # | Decision | Rationale |
|---|---|---|
| 1 | Return shape = Result-style discriminated union on `ok` | Clean TS narrowing (`if (!r.ok)`), idiomatic `Result<T,E>`, failure value is the same `TermTimeoutError` the caller would have caught. Chosen over "success unwrapped, failure discriminated on `timedOut`" (lopsided union, awkward narrowing). |
| 2 | `throws?: boolean`, **default `true`** | Back-compat. v0.2.0 callers see no change. Opt-in only. |
| 3 | Applied to **both** `waitFor` and `waitForQuiet` | Both funnel through one `poll()`; same timeout semantics. Symmetry is correct. |
| 4 | Reuse `TermTimeoutError` as the failure payload | It already has `{ code, pane, elapsed, timeout, pattern?, lastCapture }`. No new error type, no information loss. |
| 5 | No `reason` field / no combined-mode distinction | YAGNI. `lastCapture` is already on the error; the caller can diff. Revisit if usage demands. |
| 6 | **Lib-only; tool + dispatch unchanged** | The tool's `isError`-on-timeout already gives the LLM agent a structured timeout. The LLM never sees a throw, so `throws:false` adds no agent value but costs schema surface + dispatch threading + an `Error`-serialization fix. `throws:false` is for direct lib importers that want probe-then-branch without `try/catch`. |
| 7 | **Overloads** on `waitFor`/`waitForQuiet` | `(opts & { throws: false }) → WaitForResult` and `(opts) → CaptureResult` so existing `r.text` callers stay typed as `CaptureResult` (no test changes). Found during implementation: the naive `Promise<CaptureResult | WaitForResult>` return type broke existing `r.text` access; overloads restore back-compat at the type level while the runtime branches on `opts.throws`. |

## 3. Types (`lib/types.ts`)

New exported type (type-only import of `TermTimeoutError` — no runtime cycle;
`error.ts` already uses `import type` for `Pane`/`CaptureResult`):

```ts
import type { TermTimeoutError } from "./error.ts";

export interface WaitForOptions {
  timeout: number;
  quietMs?: number;
  ansi?: boolean;
  interval?: number;
  throws?: boolean;          // NEW (v0.3.0) — default true (throw on timeout)
}

export interface WaitForQuietOptions {
  ms: number;
  timeout: number;
  ansi?: boolean;
  interval?: number;
  throws?: boolean;          // NEW (v0.3.0) — default true (throw on timeout)
}

export type WaitForResult =
  | { ok: true;  result: CaptureResult }
  | { ok: false; error: TermTimeoutError };
```

## 4. Lib API (`lib/tmux.ts`)

`poll()` becomes a **pure function that never throws** — it returns
`WaitForResult`, and the public wrappers own the throw/return policy at the
boundary. This separates the decider ("did it settle?") from the policy
("throw or return on timeout") and avoids widening `poll()`'s narrow
`opts: { timeout; interval? }` type.

```ts
async function poll(pane, opts: { timeout; interval? }, done, pattern): Promise<WaitForResult> {
  // ... success: return { ok: true, result: cur }
  // ... timeout: return { ok: false, error: new TermTimeoutError(...) }  (no throw)
}

export async function waitFor(pane, pattern, opts: WaitForOptions & { throws: false }): Promise<WaitForResult>;
export async function waitFor(pane, pattern, opts: WaitForOptions): Promise<CaptureResult>;
export async function waitFor(pane, pattern, opts: WaitForOptions): Promise<CaptureResult | WaitForResult> {
  const r = await poll(pane, opts, done, pattern);
  if (r.ok) return opts.throws === false ? r : r.result;
  if (opts.throws === false) return r;   // { ok: false, error }
  throw r.error;
}
// waitForQuiet: same shape.
```

The `=== false` check keeps `undefined`/`true`/omitted on the throw path — the
default is strictly "throw", matching v0.2.0. The overloads keep the default
path typed as `CaptureResult` (existing `r.text` callers unaffected); only the
literal `{ throws: false }` call types as `WaitForResult`.

---

## 5. Pi tool surface (`extensions/term.ts`) + dispatch (`lib/dispatch.ts`)

**Unchanged.** Per Decision #6, the tool already gives the LLM agent a
structured timeout: `execute()` catches `TermTimeoutError` and returns
`{ isError: true, details: { error: "TERM_TIMEOUT", pane, elapsed, timeout, pattern } }`.
The agent never sees a throw, so `throws:false` adds no agent value. `dispatch.ts`
keeps building the wait options without `throws` (defaults to `true` → throws →
caught by the tool → structured `isError`). No schema, dispatch, or
`JSON.stringify`-normalization changes in v0.3.0.

---

## 6. Testing

All new cases are **lib-level** (no tool test — the tool is unchanged):

- `throws: false` success → `{ ok: true, result: <CaptureResult> }`; `result.text` is the captured text.
- `throws: false` timeout → `{ ok: false, error: <TermTimeoutError> }`; `error instanceof TermTimeoutError`; `error.code === "TERM_TIMEOUT"`; `error.lastCapture` is the last pane state; `error.timeout`/`error.pattern` populated.
- `throws: false` on `waitForQuiet` — same two cases (timeout path has `error.pattern === undefined`).
- Default (no `throws` field) still throws `TermTimeoutError` (regression guard).
- `throws: true` explicit — same as default (regression guard).
- Combined mode + `throws: false` — pane changes every poll (`Active ${n++}`) so it never quiets within `quietMs` → `{ ok: false, error }` (no `reason` distinction, per non-goal #2).
- `poll()` never throws — the `throws: false` timeout tests don't use `try/catch`, proving the invariant.

All unit tests use the existing mocked-exec seam. The integration smoke
(`tests/integration.test.ts`, skip-when-tmux-absent) gains one live round-trip:
a `throws: false` `waitFor` against a real pane confirms the union shape
survives a live capture — skipped under the pi overlay; run from a clean shell.

## 8. Risks & verification

| Risk | Mitigation |
|---|---|
| Return-type widening surprises a consumer that destructured `r.text` | Overloads (Decision #7) keep the default path typed as `CaptureResult`; only `{ throws: false }` widens. Existing tests unchanged. |
| `=== false` vs falsy default | Explicit `=== false` so `undefined`/`true`/omitted all throw. Unit-tested. |
| Agent forgets `throws:false` and still try/catches | Unchanged from v0.2.0 — fine; opt-in is the point. |
| pi workspace overlay discards local commits / reverts refs | reset + write + commit + **push** in ONE atomic command (no inter-command gap); verify `git ls-remote --heads origin main` after push; origin is the durable save. |
